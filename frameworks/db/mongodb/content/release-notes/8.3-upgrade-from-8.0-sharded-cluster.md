# Upgrade |oldversion| Sharded Cluster to |newversion|

.. include:: /includes/8.3-upgrade-from-8.0-replacements.rst

.. include:: /includes/minor-release.rst

Familiarize yourself with the content of this document, including thoroughly reviewing the prerequisites, prior to upgrading to MongoDB |newversion|.

The following steps outline the procedure to upgrade a :binary:`~bin.mongod` that is a shard member from version |oldversion| to |newversion|.

.. include:: /includes/upgrade-minor-versions-guidance-checklists.rst

To learn more, see `Downgrade 8.3 to 8.0 <8.3-downgrade-to-8.0>`.

## Prerequisites

### All Members Version

To upgrade a sharded cluster to |newversion|, **all** members of the cluster must be at least version |oldversion|. The upgrade process checks all components of the cluster and produces warnings if any component is running a version earlier than |oldversion|.

### Feature Compatibility Version

The |oldversion| sharded cluster must have `featureCompatibilityVersion` set to |oldfcv|.

To ensure that all members of the sharded cluster have `featureCompatibilityVersion` set to |oldfcv|, connect to each shard replica set member and each config server replica set member. Then, check the `featureCompatibilityVersion`:

> **Tip:** For a sharded cluster that has access control enabled, to run the
following command against a shard replica set member, you must
connect to the member as a :ref:`shard local user
<shard-local-users>`.

To set or update `featureCompatibilityVersion`, run the following command on the :binary:`~bin.mongos`:

```javascript
db.adminCommand( { setFeatureCompatibilityVersion: "8.0", confirm: true } )
```

For more information, see :dbcommand:`setFeatureCompatibilityVersion`.

### Replica Set Member State

For shards and config servers, ensure that no replica set member is in the :replstate:`ROLLBACK` or :replstate:`RECOVERING` state.

```javascript
db.adminCommand( { replSetGetStatus: 1 } )
```

### Back up the `config` Database

Optional but Recommended. As a precaution, create a backup of the `config` database before upgrading the sharded cluster.
